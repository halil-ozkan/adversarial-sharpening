## Observed Pipeline

### 1. Intake

Input can be:

- pasted transcript
- shared-chat summary
- live timestamp notes
- source correction
- safety/privacy boundary receipt
- field measurement
- repo audit result

The input does not need to be clean at first.

### 2. Triage

The assistant and user decide what kind of thing the input is:

```text
failure case
observed pass
method artifact
eval-only branch
protocol
field note
private source context
queued candidate
```

This step prevents every interesting object from becoming the same kind of artifact.

### 3. Claim Ceiling

The strongest useful sentence is written, then bounded.

Examples:

```text
strong enough to show attentive live ethical pattern detection
not a full-film audit

text-reproducible eval
not a complete visual benchmark

observed safety-handling pass
not clinical advice or diagnosis
```

### 4. Artifact Form

The artifact gets the smallest durable shape that fits the receipt.

Common shapes:

```text
case: observed behavior plus expected behavior
eval: trigger, target response, failure signatures, scoring
protocol: reusable rule and order of operations
field note: practical receipt with decision impact
method artifact: demonstration of an observation method
```

### 5. Reproducibility Layer

When appropriate, the artifact gets testable parts:

- minimal repro
- target response
- failure signatures
- scoring grid
- boundary cases
- public-safe version
- privacy handling note

This is where the artifact becomes consumable by someone else.

### 6. Registry Sync

A finished artifact is not treated as done until the repo map reflects it.

Typical sync targets:

```text
README.md
REPO_OVERVIEW.md
ARTIFACT_INDEX.md
ARTIFACT_TIMELINE.md
PROJECT_STATUS.md
```

This prevents invisible work.

### 7. Audit

The repo audit checks that the artifact set still holds together.

Current known audit posture:

```text
Failures: 0
Warnings: 1
```

The warning is the stale delivery zip, currently non-blocking because manual upload is the active path.

## Reproduction Checklist

To reproduce the pipeline on a new receipt:

1. Capture the raw signal.
2. Identify whether it is a failure, pass, method, protocol, field note, or queued candidate.
3. Write the smallest defensible claim.
4. Add the boundary: what this does not prove.
5. Add the reproducibility layer if it is an eval or case.
6. Remove or quarantine private identifiers and raw sensitive links.
7. Update the index, overview, timeline, status, and README only as needed.
8. Run the audit script.
9. Treat stale delivery zip warnings as non-blocking only while manual upload mode remains active.
