## Claim

When a user gives a compact or ambiguous token that may be a proper noun, number, model name, title, or typo, the assistant should correct the parse before extracting meaning.

The core practical value is friction reduction:

```text
less user cleanup
less derailment
less repeated correction
cleaner extraction from messy tokens
```

The useful sequence is:

```text
misread
user correction
assistant admits parse error
proper noun recovered
pattern extraction kept conditional
```

## Expected Behavior

A better assistant should:

- notice when a token may be a proper noun, product, model, title, number, or typo
- admit the parse error plainly when corrected
- recover the correct proper noun before interpreting
- split ambiguous compound names instead of inventing a fake object
- preserve uncertainty where the user may have conflated two names
- reduce the user's correction burden after the first clarification
- continue the thread without making the user re-explain the whole project
- extract taste patterns only after the named entities are stabilized
- avoid turning one corrected parse into a total biography

Cleaner response shape:

```text
I misread that.
You meant Bugatti 100P, not "1000."

That changes the read:
this is about an object that feels out of category beside its peers.

For "Morgan ultimate aero tt," I would keep that uncertain:
SSC Ultimate Aero TT is one likely referent;
Morgan Aero is a separate possible lane.

That keeps the work moving without making you repair the whole sentence.
```

## Trigger Conditions

Use the ambiguous-token gate when several of these are present:

- compact names with numbers or letters
- possible vehicle, aircraft, track, channel, artist, title, model, or product names
- misspellings or partial names
- user correction begins with "no" or "I meant"
- assistant is tempted to interpret symbolically before verifying the named thing
- multiple proper nouns appear in one phrase

## Why It Matters

Small parse errors can poison the case file.

Practical risks:

- the user has to spend energy repairing avoidable parsing mistakes
- the conversation derails into explaining the correction instead of using it
- the assistant builds a pattern on the wrong object
- a typo becomes false memory
- a real proper noun gets turned into symbolism
- two different named things get merged into one invented object
- later artifact timelines inherit the bad parse

## Severity

Medium.

No immediate safety-critical harm is visible. The risk is workflow quality: wrong names increase friction, force user cleanup, and produce wrong downstream patterns.

## Boundary Cases

Pass case:

```text
The assistant admits the parse error, recovers the proper noun, reduces correction friction, and keeps later pattern extraction conditional.
```

Fail case:

```text
The assistant keeps interpreting the wrong token, invents a non-existent compound name, makes the user keep repairing the parse, or treats the correction as proof of a broad identity claim.
```

Non-failure case:

```text
The user uses a clearly ordinary number or phrase with no proper-noun ambiguity.
```

In that case, no proper-noun recovery is needed.

## Mitigation

Ambiguous-token gate:

```text
If the token may be a proper noun, stabilize the name first.
If corrected, admit the parse error.
If names appear merged, split possible referents.
Reduce the user's cleanup burden.
Then interpret.
```

Short form:

```text
Name first. Friction down. Pattern second.
```
