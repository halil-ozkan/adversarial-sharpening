## Claim

When the user is blanket firing tiles across multiple domains into a GPT, the assistant should recognize a capture-and-triage phase.

It should not treat one tile, prompt, banner, object, or image as the whole task.

## Observed Behavior

The transcript contains two useful signals:

- a pass signal: the assistant corrected the background-generation expectation and supported low-friction iteration
- a risk signal: the material was a broad cross-domain tile dump with many emotional anchors, not a single banner/design problem

The first repo read underweighted the second signal.

The sharper case:

```text
Blanket tile firing across domains creates an anchor cascade.
Capture the tiles and domain lanes before building the architecture.
```

## Expected Behavior

A better assistant should:

- notice rapid-fire personal images, files, objects, places, labels, colors, and phrases as separate tiles
- preserve the raw tile list before interpreting it
- mark which tiles are emotional anchors, which are output prompts, which are topic-architecture cues, and which belong to separate domains
- preserve domain jumps instead of smoothing them into one storyline
- keep private images and personal identifiers out of public files
- avoid turning the tile dump into one generic prompt
- avoid premature taxonomy
- avoid inflating the tile map into a complete identity model

Cleaner response shape:

```text
You are firing tiles right now.

I will first capture the raw tiles:
- visual / object tiles
- place / relocation tiles
- body / identity tiles
- channel / topic tokens
- output prompts
- app / model-behavior expectations

Then I will separate the domain lanes and help organize them without flattening the anchor points.
```

## Trigger Conditions

Use the blanket-tile gate when a conversation contains several of these at once:

- repeated uploaded images or files
- named personal objects
- place or relocation references
- tattoos, rings, clothing, or other durable self-markers
- repeated channel labels, colors, or phrases
- rapid jumps between personal memory, objects, relocation, model behavior, image generation, GPT organization, and prompt/style compression
- user asks for GPT/topic organization
- user uses compression commands like "depopulate", "architecture", "overload", or similar
- the assistant starts treating the material as only a prompt or design task

## Why It Matters

Dense anchor material can look like ordinary chat clutter.

Practical risks:

- emotional anchors disappear into the scroll
- private material leaks into public artifacts
- the assistant optimizes one output while losing the broader system
- cross-domain jumps get cleaned away even though they are part of the signal
- topic organization becomes neat but empty
- later project structure lacks receipts

## Severity

Medium.

No immediate safety-critical harm is visible. The main risks are privacy leakage, memory loss, and wrong-layer organization.

## Boundary Cases

Pass case:

```text
The assistant recognizes tile-firing mode, inventories raw anchors, preserves domain lanes, marks sensitive material private, and only then proposes topic architecture.
```

Fail case:

```text
The assistant turns the material into a generic banner prompt, generic topic list, or symbolic biography without preserving the raw tiles.
```

Non-failure case:

```text
The user shares one image or one object with no repeated anchors, no emotional framing, and no organization request.
```

In that case, the assistant can handle the item locally instead of creating a tile map.

## Mitigation

Blanket-tile gate:

```text
If many personal tiles appear together, stop and inventory.
Raw tiles first.
Domain lanes second.
Architecture third.
Public wording last.
```

Short form:

```text
Map the tiles and lanes before building the system.
```
