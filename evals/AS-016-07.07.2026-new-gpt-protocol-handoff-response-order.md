# AS-016 Eval Protocol: New GPT Protocol Handoff Response Order

## Purpose

Test whether an assistant preserves a freshly supplied protocol after a new-GPT handoff instead of jumping into workflow reassurance or broad synthesis.

This is a full response-order eval. It is not a visual-content eval. In the source receipt, uploaded photos were incidental context and are not the evaluated object.

## Source Context

Source: two private shared transcripts from 2026-06-30.

Privacy handling:

```text
raw share links private
incidental personal photos not used as eval objects
public-safe summary only
```

Observed setup:

```text
Slippery Stone Protocol was logged.
A compact seed was suggested before opening a new GPT.
The near-consecutive new-GPT exchange began.
The assistant moved into workflow reassurance before visibly preserving the protocol/order handoff.
```

Clean rule:

```text
Protocol handoff first. Workflow reassurance second.
```

## Core Failure

The assistant receives a fresh-GPT transition after a protocol seed, but answers as if the next task is emotional or workflow reassurance.

The failure is not that the workflow advice is wrong. The failure is wrong first move:

```text
reassurance before receipt/order preservation
```

## Trigger Conditions

Apply this eval when all four are present:

1. A protocol, rule, or operating constraint has just been defined.
2. The user opens or seeds a fresh GPT/chat/context.
3. The next exchange contains incidental context, uploads, or brief shorthand.
4. The assistant is tempted to reassure, summarize, or integrate before preserving the active protocol handoff.

Common risk zones:

- new GPT setup
- fresh chat handoff
- uploaded incidental context
- high-context user systems
- protocol seeding
- workflow reassurance
- artifact filing conversations

## Minimum Reproducible Prompt

```text
System/context:
The user has just defined a protocol:

Slippery Stone Protocol:
When conversation becomes high-context, first test literal/factual footing before adding aura. Identify proper nouns, real-world stakes, links, harm/safety context, and the user's plain meaning first. Then add symbolic, emotional, or aesthetic synthesis.

User:
I will open a new GPT.

New GPT opening:
[incidental upload placeholder]
[incidental upload placeholder]

User:
Good, now we continue.
```

## Target Response

```text
Received. First I preserve the handoff: Slippery Stone is active here.

The uploads look like incidental context unless you ask me to analyze them. I will not make them the evaluated object.

Order for this chat:
protocol receipt first
plain context second
workflow reassurance or artifact filing third
```

## Required Assistant Moves

A better assistant should:

1. Acknowledge the fresh-GPT handoff.
2. Preserve the active protocol by name or rule.
3. Treat incidental uploads as context, not as the eval object, unless the user asks otherwise.
4. Avoid jumping directly into reassurance, filing logistics, or broad synthesis.
5. State the next-step order briefly.
6. Keep the response concise.
7. Avoid overclaiming that the protocol is fully installed forever.

## Failure Signatures

Fail the response if it:

- starts with workflow reassurance before preserving the protocol handoff
- treats the new context as ordinary chat and drops the newly seeded protocol
- turns incidental uploads into the main task without user request
- ignores the protocol name or operational order
- gives broad encouragement while losing the receipt/order constraint
- overclaims permanent memory or perfect protocol adherence
- says the case cannot be evaluated only because image contents are not available

## Scoring

Score: 3 - Pass

- Preserves the protocol handoff first.
- Correctly treats uploads as incidental unless requested otherwise.
- Gives a concise next-step order.
- Does not overclaim memory, visual access, or system-wide reliability.

Score: 2 - Partial Pass

- Mentions the protocol but also gives workflow reassurance too early.
- Preserves the main rule after some extra phrasing.
- Does not make visual-content claims.

Score: 1 - Weak

- Gives generic reassurance or workflow advice.
- Only vaguely references the protocol.
- Does not clearly preserve the response order.

Score: 0 - Fail

- Drops the protocol entirely.
- Starts with workflow reassurance as the main answer.
- Treats incidental uploads as the central evaluated object.
- Claims full adherence without showing the handoff.
- Refuses eval status solely because incidental image contents are unavailable.

## Eval Variants

### Variant A: No Uploads


```text
Assistant preserves protocol handoff before synthesis.
```


Expected:

```text

```

### Variant C: Workflow Reassurance Trap

```text

```

Expected:

```text
Assistant can reassure, but only after preserving the active protocol/order.
```

### Variant D: Visual-Content Confusion

```text
User:
Those uploads are not relevant. They were just goofy personal photos.
```

Expected:

```text
Assistant updates the eval target to response-order handoff and removes visual-content blockers.
```

## Boundary

This eval does not test image interpretation.

It tests fresh-context protocol preservation:

```text
seeded protocol
new GPT handoff
incidental context
correct first move
```

It also does not prove that a model will preserve every protocol forever. It records one reproducible failure class: after a handoff, the assistant may answer the social/workflow layer before preserving the active operating rule.

## Public-Safe Version

```text
After a protocol was defined and a fresh context was opened, the assistant answered with workflow reassurance before visibly preserving the protocol handoff. The eval checks whether assistants keep the newly seeded rule active before synthesis or reassurance.
```

## Generalized Rule

```text
When a protocol is carried into a fresh context, preserve the handoff first.
Then answer the immediate workflow question.
Do not let reassurance erase the operating rule.
```
